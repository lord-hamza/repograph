## 1. Product Overview

RepoGraph is a developer tool that ingests any GitHub repository and produces two outputs: an interactive visual dependency graph of the entire codebase, and a structured MCP context file loadable into Claude Code or any compatible AI agent. It ships as three surfaces — a CLI tool (`npx repograph`), a web platform (`repograph.com/[owner]/[repo]`), and a GitHub Action for CI-integrated auto-generation.

The core value proposition is codebase transparency — making the invisible structure of any repo instantly visible to both humans and AI agents.

---

## 2. Problem Statement

### For Developers
Large codebases are opaque by default. A developer joining a new project must manually trace imports, read through directories, and build a mental model over days or weeks. This cost is incurred repeatedly — on every onboarding, every new contractor, every code review.

### For AI Coding Assistants
Tools like Claude Code and Cursor operate file-by-file. They read what is open in the editor. They have no structural awareness of how that file relates to the rest of the codebase — what imports it, what it depends on, where it sits in the dependency chain. This limits the quality of refactoring suggestions, the accuracy of impact analysis, and the relevance of generated code.

### For Code Review
Reviewers approving PRs frequently lack visibility into what a change breaks downstream. Without a dependency map, the blast radius of any change is invisible.

### For Documentation
Documentation is written once and goes stale immediately. It never reflects live structure. Autogenerated docs from comments are marginally better but still don't show cross-module relationships.

RepoGraph solves all four of these with a single scan.

## 3. Solution Architecture

RepoGraph is built as a monorepo with a shared core engine consumed by three separate surfaces.

```
repograph/
├── packages/
│   └── core/           Published npm package — the parser and graph engine
├── apps/
│   ├── cli/            npx repograph CLI
│   ├── web/            Next.js web platform (repograph.com)
│   └── worker/         BullMQ async scan worker
└── action/             GitHub Action definition
```

### The Engine-First Philosophy

The core engine (`repograph-core`) is the only logic layer. All three surfaces (CLI, web, GitHub Action) are thin consumers of this package. This means:

- The graph output is deterministic and testable independent of any UI
- The CLI and web always produce identical structural analysis
- Updates to parsing logic propagate everywhere simultaneously

### Two-Output Model

Every scan produces exactly two outputs:

**Output 1 — Graph JSON**: A normalized node-edge structure representing the entire repo. Nodes are files, functions, classes, and modules. Edges are imports, calls, and inheritance relationships. This is consumed by the D3 visualization.

**Output 2 — MCP Markdown**: A structured context document formatted for loading into Claude Code as an MCP server. It describes architecture, entry points, module relationships, tech stack, and dependency chains in a form optimized for language model context injection.

## 4. System Components

### 4.1 repograph-core (The Engine)

The engine is a TypeScript npm package. It has no UI dependencies and no network dependencies beyond optional GitHub API calls for remote ingestion.

**Ingestion Layer**
Handles two input modes: a GitHub repository URL (uses GitHub REST API to fetch file tree and raw file contents) and a local filesystem path (direct fs traversal). Both normalize to the same internal file-content map before parsing begins. For GitHub URLs, the engine respects rate limits, handles pagination for large repos, and caches file content to minimize API calls.

**Parser Layer**
Language-specific parsers run against each file. Each parser extracts: file path, all import/require/export statements, all function definitions (name, signature, line range), all class definitions (name, methods, inheritance), and all cross-module references. Parsers are language-pluggable — new language support is added by registering a new parser without touching the rest of the engine.

Python parsing uses `tree-sitter` with the Python grammar. This handles all Python syntax including decorators, async functions, dataclasses, and type annotations.

JavaScript and TypeScript parsing uses the TypeScript Compiler API directly. This handles JSX, TSX, ES modules, CommonJS, dynamic imports, and TypeScript-specific constructs like interfaces, enums, and type aliases.

**Graph Builder**
After all parsers run, the graph builder performs resolution. It takes the raw extracted data (imports that are strings like `"../utils/format"`) and resolves them to actual file nodes in the graph. It handles relative paths, index file resolution, node_modules exclusion, and circular dependency detection. The output is a normalized graph where every edge points to a real node.

**Tech Stack Detector**
A fingerprinting layer that runs on the completed graph and file manifest. It identifies frameworks, libraries, databases, and tools by combining: `package.json` / `requirements.txt` / `pyproject.toml` dependency analysis, import frequency across the codebase, configuration file presence (webpack, vite, prisma, etc.), and structural patterns (e.g., presence of `routes/` directory with Express imports = Express.js routing layer). For each detected technology, it records: what it is, which files use it, how many times it appears, and what other technologies it interacts with.

**Exporters**
Two exporters consume the completed graph:

The JSON Exporter serializes the graph model to a flat JSON structure with nodes array, edges array, metadata (repo name, scan timestamp, commit SHA, language breakdown, file count, dependency depth), and tech stack annotations. This is the format consumed by D3 and stored in Redis/Postgres.

The MCP Markdown Exporter converts the graph into a structured Markdown document. It produces sections for: project overview, directory structure, entry points, module map (each module with its exports and dependencies), tech stack summary with usage locations, and critical path analysis (the most-imported files in the repo). This document is formatted specifically for injection into Claude Code's MCP context window.

### 4.2 CLI (apps/cli)

The CLI is a Node.js command-line tool distributed via npm and runnable with `npx repograph`.

**Command Structure**

`repograph pull <repo-url-or-path>` is the primary command. It accepts a GitHub URL (https or SSH), a local directory path, or `.` for current directory. Optional flags include `--output` to specify output directory, `--format` to choose json/md/both, `--depth` to limit dependency traversal depth, and `--open` to auto-open the graph in browser.

**Terminal Output**
After parsing completes, the CLI prints a structured summary to the terminal: repository name and scan timestamp, language breakdown (files per language), total nodes and edges in the graph, most-imported files (top 10), tech stack detected, circular dependencies detected (if any), and output file paths.

**Graph Output**
The CLI generates a self-contained HTML file containing the D3 visualization with the graph data embedded inline. No server required. The file opens in the default browser. This HTML file is shareable — it runs standalone with no external dependencies.

**MCP File Output**
The MCP Markdown file is written to the current directory as `repograph-context.md`. The CLI prints instructions for loading it into Claude Code.

### 4.3 Web Platform (apps/web)

The web platform runs at `repograph.com`. The URL scheme mirrors GitHub: `repograph.com/[owner]/[repo]` maps directly to the GitHub repo `github.com/[owner]/[repo]`. The implicit user flow is a browser URL edit — change `github.com` to `repograph.com` and land on the analysis.

**Page Architecture**

The page is a single-page experience with four panels. All panels populate progressively as the scan streams results. There is no loading screen blocking access — the UI renders in skeleton state and fills in as data arrives.

**Panel 1 — Dependency Graph**
A force-directed interactive graph rendered in D3. Nodes are colored by type (file, class, function) and by module/directory cluster. Edges show import direction. Interactive behaviors include: click a node to highlight its immediate dependencies and dependents, drag nodes to rearrange, zoom and pan, search/filter by node name, and toggle edge types (imports only, calls only, all). The graph is the hero element of the page — it is designed to be screenshot-worthy. Node clustering uses a force simulation that naturally groups related modules together, making architectural boundaries visually apparent.

**Panel 2 — Directory Tree**
A collapsible directory tree showing the full repo structure. Each file in the tree is linked to its node in the graph — clicking a file in the tree highlights it in the graph and vice versa. The tree shows file type indicators, import count per file (how many other files import this one), and a color-coded dependency depth indicator.

**Panel 3 — Tech Stack Explorer**
A grid of detected technologies, each as a clickable card showing: technology name, category (framework, database, auth, testing, etc.), file count using it, and a usage percentage badge. Clicking any technology opens a side drawer. The drawer shows: what the technology is (one-sentence description), why it's used in this repo (inferred from context), where it's used (every file that imports or references it, clickable to jump to that node in the graph), and how it connects to other technologies (e.g., "Prisma connects to PostgreSQL and is called from 14 API route handlers"). The drawer essentially traces the full journey of that technology through the codebase.

**Panel 4 — MCP Download**
A prominent download section with a single button: Download MCP Context File. The downloaded file is the MCP Markdown output from the engine. Below the button, brief instructions explain how to load it into Claude Code. A secondary option copies a one-line Claude Code command to clipboard that loads the file directly.

**Scan Progress**
When a repo is not yet cached, the UI shows a live scan progress feed via Server-Sent Events. Progress stages shown: fetching file tree, parsing files (with a live counter), building dependency graph, detecting tech stack, generating outputs. This feed is the "futuristic UI" moment — watching the codebase being understood in real time.

---

### 4.4 Worker (apps/worker)

Long-running scan jobs cannot execute inside Vercel's serverless functions due to timeout limits. The worker is a separate Node.js process running on Railway that processes the scan queue.

When a web request arrives for an uncached repo, the API enqueues a job to BullMQ. The worker picks up the job, runs the full `repograph-core` scan pipeline, stores results in Redis (for fast retrieval) and Postgres (for persistence and analytics), and emits progress events via SSE that the web client receives in real time. If a scan fails, the worker retries up to three times with exponential backoff before marking the job as failed and returning an error state to the client.

---

### 4.5 GitHub Action (action/)

A YAML-defined GitHub Action that runs `repograph-core` on every push to the default branch. It generates the graph HTML and MCP context file and commits them to the `gh-pages` branch, making the live graph available at `[owner].github.io/[repo]/repograph`. It also posts a comment on any PR with a link to the graph diff (what dependencies changed in this PR). Configured via a simple `repograph.yml` in the repo's `.github/workflows/` directory.

## 5. Feature Specifications

### F1 — Repo Ingestion
- Accept public GitHub repository URL (HTTPS and SSH format)
- Accept local filesystem path
- Accept `.` for current working directory (CLI only)
- Support mono-repos (detect and parse all packages)
- Exclude `node_modules`, `.git`, `dist`, `build`, `__pycache__` by default
- Configurable exclusion patterns via `.repographignore` file
- Handle repos up to 50,000 files
- Handle binary files gracefully (skip, do not fail)

### F2 — Language Support (v1)
- Python (3.x)
- JavaScript (ES6+, CommonJS)
- TypeScript (including JSX/TSX)
- MERN stack projects (full support)

### F3 — Graph Generation
- File nodes with metadata (path, language, line count, import count)
- Function nodes with metadata (name, file, line range, call count)
- Class nodes with metadata (name, file, methods, inheritance)
- Import edges (file A imports file B)
- Inheritance edges (class A extends class B)
- Circular dependency detection and flagging
- Orphan file detection (files with no imports and no importers)
- Entry point detection (files with no importers but many dependents)

### F4 — Tech Stack Detection
- Detect frameworks, libraries, databases, ORMs, auth libraries, testing tools, build tools
- Map each technology to the files that use it
- Identify technology interaction pairs
- Confidence scoring per detection
- Support for Python and Node.js ecosystems in v1

### F5 — MCP Context Generation
- Structured Markdown output
- Sections: overview, architecture summary, directory structure, module map, tech stack, entry points, critical files, dependency chains
- Optimized token density for Claude Code context window
- Version-stamped (includes commit SHA)
- Human-readable without any tooling

### F6 — CLI
- `repograph pull <target>` primary command
- Terminal stats output
- Self-contained HTML graph output
- MCP Markdown output
- `--watch` mode for local development (re-scan on file change)
- `--diff <commit-sha>` to show dependency changes between commits

### F7 — Web Platform
- URL routing: `repograph.com/[owner]/[repo]`
- Cache-first loading (instant for previously scanned repos)
- SSE-based live scan progress for uncached repos
- Interactive D3 force graph
- Directory tree with graph cross-linking
- Tech stack drawer with full journey tracing
- MCP context file download
- Share button generating a permalink to the current graph view
- GitHub OAuth for private repo support (Phase 2)

### F8 — GitHub Action
- Auto-scan on push to default branch
- Publish graph to GitHub Pages
- PR comment with dependency diff
- Badge generation for README

## 6. Data Models

### GraphNode
Every entity in the codebase is a node. Each node has a unique ID (file path + entity name), a type (file / function / class / module), a label (display name), metadata (language, line count, file path), and computed metrics (in-degree, out-degree, centrality score).

### GraphEdge
Every relationship is an edge. Each edge has a source node ID, a target node ID, a type (import / inheritance / call / export), and a weight (how many times this relationship appears).

### RepoGraph
The top-level data structure. Contains: repository metadata (name, owner, default branch, commit SHA, scan timestamp), a nodes array, an edges array, language breakdown statistics, tech stack detections, entry points list, and orphan files list.

### TechStackEntry
Each detected technology has: name, category, confidence score, list of files using it, list of technologies it interacts with, and a usage count.

### ScanJob
Queue job record with: job ID, repo identifier, status (queued / scanning / complete / failed), progress stage, error message (if failed), result cache key (if complete), and timestamps.

## 7. Flows

### 7.1 Web Scan Flow — Cache Hit

User navigates to `repograph.com/vercel/next.js`. Next.js server checks Redis for cached graph keyed by `[owner]/[repo]/[latest-commit-sha]`. Cache hit returns graph JSON immediately. Page renders with full graph, directory tree, and tech stack populated within one network round trip. No worker involvement.

### 7.2 Web Scan Flow — Cache Miss

User navigates to `repograph.com/facebook/react`. Next.js server checks Redis — no cache entry. API route creates a scan job in BullMQ and returns a job ID with SSE stream endpoint. Client connects to SSE stream. Worker picks up job, begins repo ingestion via GitHub API, emits progress events (file fetching, parsing, graph building, exporting) via Redis pub/sub to SSE endpoint. Client receives events and animates the progress feed in the UI. On completion, worker stores graph in Redis and Postgres. SSE sends a `complete` event with the graph data. Client renders all four panels. Total time for a mid-sized repo (5,000–15,000 files): 15–45 seconds.

### 7.3 CLI Flow

Developer runs `npx repograph pull https://github.com/owner/repo` in VSCode terminal. Tool fetches file tree via GitHub API. Parses all supported files. Builds graph. Prints stats table to terminal. Writes `repograph-graph.html` and `repograph-context.md` to current directory. Prints file paths and a one-line Claude Code command. If `--open` flag used, opens graph HTML in default browser. Total time for same mid-sized repo: 10–30 seconds.

### 7.4 MCP Loading Flow

Developer downloads `repograph-context.md` from the web UI or generates it via CLI. Adds it to their Claude Code project context via the MCP configuration. Claude Code reads the context file at session start. All subsequent prompts within that session have full structural awareness of the repo — Claude Code knows the entry points, the module relationships, the tech stack, and the critical files before reading a single line of application code.

### 7.5 GitHub Action Flow

Developer merges PR to main. GitHub Action triggers. Action checks out repo, runs `repograph-core`, generates graph HTML and MCP context file. Commits outputs to `gh-pages` branch. Posts comment on the merged PR with a link to the live graph and a diff summary showing which dependency edges were added or removed by this merge.

### 7.6 Tech Stack Drawer Flow

User is on `repograph.com/owner/repo`. Panel 3 shows tech stack grid. User clicks "Prisma" card. Drawer slides in from the right. Drawer header shows Prisma name, version detected, category (ORM), and confidence score. Body shows: description (one sentence), a "Used in N files" section listing each file as a clickable chip that focuses that node in the graph when clicked, an "Interacts with" section showing PostgreSQL and the API route layer, and a "Dependency chain" mini-graph showing Prisma's position in the tech stack hierarchy. User clicks a file chip. Drawer stays open. Graph panel pans and highlights the selected node. User can navigate the entire tech stack without leaving the page.

## 8. Tech Stack

### Core Engine (packages/core)

**Language:** TypeScript 5.x  
All engine code is TypeScript. Shared types between web, CLI, and worker. Strong typing on the graph model prevents data shape drift across surfaces.

**Python Parsing:** tree-sitter with tree-sitter-python grammar  
Production-grade incremental parser. Handles all Python 3.x syntax. Fast enough for large repos without streaming. Well-maintained with an active grammar.

**JS/TS Parsing:** TypeScript Compiler API  
Native TypeScript understanding. Handles JSX, TSX, decorators, module resolution, type-only imports. More accurate than regex-based or Babel-based alternatives for complex TS projects.

**Graph Library:** graphology  
Lightweight, serializable graph data structure. Supports directed multigraphs. Has built-in algorithms for centrality, shortest path, and connected components — useful for the critical path and orphan detection features.

**Package Distribution:** npm (published as `repograph-core`)  
Consumed by CLI and worker directly. Versioned independently.

---

### CLI (apps/cli)

**CLI Framework:** commander.js  
Standard, well-documented. Handles argument parsing, help generation, and subcommands.

**Terminal UI:** ora (spinners) + chalk (colors) + cli-table3 (stats table)  
Standard combination for polished terminal output without heavy dependencies.

**Graph HTML Output:** Self-contained D3.js bundle embedded in template HTML  
No server needed. The output file is drag-and-drop into any browser.

**Distribution:** `npx repograph` via npm  
Zero install. Single command. Works on macOS, Linux, Windows.

---

### Web Platform (apps/web)

**Framework:** Next.js 15 (App Router)  
Server components for initial page render and SEO. Client components for interactive panels. API routes for scan job creation and SSE streaming. Chosen per team preference.

**UI Library:** shadcn/ui  
Component primitives. Drawer, Card, Badge, Tooltip, Skeleton components used throughout. Fully customizable — no fighting with library styles.

**Styling:** Tailwind CSS v4  
Chosen per team preference. Utility-first, fast iteration on futuristic dark UI aesthetic.

**Graph Visualization:** D3.js (v7) — force-directed simulation  
Full control over node rendering, force parameters, zoom behavior, and interaction. No wrapper library that limits customization. The graph is a key UI differentiator — D3 gives the most control over making it look exceptional.

**Progress Streaming:** Server-Sent Events (SSE)  
One-way server-to-client stream. Simpler than WebSockets for the scan progress use case. Native browser support. Works through Vercel edge functions.

**Job Queue:** BullMQ  
Redis-backed job queue. Handles retry logic, job prioritization, and worker concurrency. Standard choice for async processing in Node.js.

**Cache:** Redis (Upstash)  
Graph JSON cached keyed by `[owner]/[repo]/[commit-sha]`. TTL set to 7 days. Upstash chosen for serverless-compatible Redis (HTTP-based, no persistent connection required from Vercel).

**Database:** PostgreSQL via Supabase  
Persists: scan history, repo metadata, usage analytics, user accounts (Phase 2). Supabase chosen for built-in auth, REST API, and managed Postgres.

**Auth:** Clerk  
GitHub OAuth for private repo access (Phase 2). Clerk chosen for simplicity and first-class Next.js App Router support.

**Hosting — Web:** Vercel  
Next.js native. Handles SSR, API routes, and static assets. Auto-deploys on push.

**Hosting — Worker:** Railway  
Long-running Node.js process. BullMQ workers cannot run on Vercel's serverless model. Railway gives persistent containers with simple deployment.

---

### GitHub Action (action/)

**Runtime:** Node 20  
Direct execution of `repograph-core` without any additional runtime.

**Output Publishing:** `peaceiris/actions-gh-pages`  
Standard action for committing to `gh-pages` branch.

---

### Monorepo Management

**Turborepo**  
Task pipeline management (build, test, lint run in correct dependency order across packages). Caching of build outputs. `packages/core` must always build before `apps/cli`, `apps/web`, or `apps/worker`.

**Package Manager:** pnpm  
Workspace support. Faster installs. Correct handling of shared `node_modules` in monorepo context.

## 9. Non-Functional Requirements

### Performance
- CLI scan of a 10,000-file repo must complete in under 60 seconds on a standard developer laptop
- Web scan of a 10,000-file repo must complete within 60 seconds on the worker
- Cached graph pages must achieve Time to Interactive under 2 seconds
- D3 graph must handle up to 5,000 nodes without frame drops on a modern laptop
- For graphs over 5,000 nodes, progressive LOD (level of detail) rendering is applied — distant nodes are simplified

### Scalability
- Worker pool must scale horizontally — multiple worker instances consume the same BullMQ queue
- Redis cache must handle concurrent requests for the same repo (only one scan job created per repo per commit SHA, regardless of concurrent requests)
- GitHub API rate limit must be respected — authenticated requests (5,000/hour) used for all web scans

### Reliability
- Scan failures must not crash the worker — failed jobs are retried 3 times, then marked failed with a user-visible error
- Partial graph results (incomplete scan due to API limits) must be surfaced as a warning, not silent data loss
- CLI must work fully offline for local path inputs with no network dependency

### Accuracy
- Import resolution must correctly handle relative paths, aliases (tsconfig paths, webpack aliases), and index files
- Tech stack detection must achieve >90% precision on common stacks (React, Express, FastAPI, Django, Next.js, NestJS)
- False positives in tech detection are worse than false negatives — prefer conservative detection

### Developer Experience
- `npx repograph pull .` must work in a fresh terminal with no configuration
- Web UI must explain what it is and how to use it without documentation — the UI is self-describing
- MCP file must work with Claude Code without any manual editing

## 10. Infrastructure & Deployment

### Production Environment

```
Vercel (web app)
  └── connects to → Upstash Redis (cache + pub/sub)
  └── connects to → Supabase Postgres (persistence)
  └── enqueues to → Upstash Redis (BullMQ queue)

Railway (worker)
  └── reads queue from → Upstash Redis
  └── writes results to → Upstash Redis + Supabase Postgres
  └── calls → GitHub API
```

### Environments
Three environments: development (local), staging (preview deployments on Vercel), and production. Environment variables managed via Vercel environment config and Railway service variables.

### Monitoring
Upstash console for Redis metrics and queue depth. Railway metrics for worker CPU/memory. Vercel analytics for web traffic. Sentry for error tracking across web and worker. Custom dashboard (simple Next.js admin page) showing scan volume, cache hit rate, and worker queue depth.

### CI/CD
GitHub Actions pipeline: on every PR, runs lint, type-check, and unit tests across all packages via Turborepo. On merge to main, Vercel auto-deploys web. Railway auto-deploys worker via GitHub integration.

## 11. Security & Access

### Public Repos
No authentication required. GitHub API called with a shared service token (authenticated to get 5,000 req/hour). Rate limit shared across all unauthenticated web users, managed by the worker.

### Private Repos
Phase 2 feature. Requires GitHub OAuth via Clerk. User grants repo access. Access token stored encrypted in Supabase, scoped per user. Scans of private repos are user-private — results not cached in shared Redis, stored only in user-scoped Postgres rows.

### CLI and Local Paths
Local filesystem scans never leave the machine. No data transmitted. For GitHub URLs via CLI, the GitHub API is called directly from the developer's machine using their own network — no RepoGraph backend involvement unless they choose to share.

### Data Retention
Cached graphs in Redis expire after 7 days. Postgres records are retained indefinitely for analytics. Users can request deletion of any scan associated with their account.

### API Security
All web API routes validate input (repo URL format, owner/repo length limits) to prevent injection. GitHub API calls are made server-side only — no GitHub tokens exposed to the client. BullMQ jobs include a HMAC signature to prevent job forgery.

## 12. Milestones & Build Order

### Milestone 1 — Engine (Weeks 1–2)
Build `repograph-core` with Python and TypeScript parsers, graph builder, tech stack detector, and both exporters. Validate on 5 real-world repos of varying size and complexity. Unit test the graph resolution logic thoroughly.

### Milestone 2 — CLI (Weeks 2–3)
Build the CLI on top of the engine. Ship to npm. Test on macOS, Linux, and Windows. The CLI is the first public artifact — it can get GitHub stars before the web platform exists.

### Milestone 3 — Web Platform (Weeks 3–6)
Build the Next.js web platform. Worker first (validate async scan pipeline), then the four UI panels in order of complexity: directory tree, graph, tech stack drawer, MCP download. Polish the scan progress animation last.

### Milestone 4 — GitHub Action (Week 6–7)
Wrap the engine in an Action. Test with a variety of real repos. Publish to GitHub Marketplace.

### Milestone 5 — Private Repo Support (Phase 2)
Add GitHub OAuth, user accounts, and private repo scanning. This is a growth feature — free tier is public repos only.
